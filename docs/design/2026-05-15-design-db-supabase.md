## 约束条件

- **Supabase 免费版：最多 2 个 project**
- 决策：全部数据放入 1 个 project（`payment-playground`），第 2 个 project 留备用
- 隔离方式：**PostgreSQL 多 schema**，每个 app 独占一个 schema

---

## Project 分配

| Project | 用途 |
|---------|------|
| `payment-playground` | 全部应用数据：demo-hub、admin-console、所有电商站 |
| 备用 | 暂留（未来可作 staging 环境或独立业务） |

---

## Schema 总览

```
payment-playground
├── auth.*              ← Supabase 内置，所有 app 用户共用 auth.users
├── public.*            ← 默认 schema，暂留共享工具函数
├── demohub.*           ← demo-hub 产品展示配置
├── admin.*             ← admin-console 管理配置
├── fashion.*           ← store-fashion 电商数据
└── (future: electronics.*, ai_subscription.*, ...)
```

**Supabase Dashboard 操作：** Settings → API → Exposed schemas，开启 `demohub`、`admin`、`fashion` 等。

**JS 客户端用法：**
```js
supabase.schema('demohub').from('products').select('*')
supabase.schema('fashion').from('orders').select('*')
```

## Schema: `demohub`

> 原 `public.demo_hub_products` 迁移至此。

### 表：`demohub.products`

```sql
CREATE SCHEMA IF NOT EXISTS demohub;

CREATE TABLE demohub.products (
  id           uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  provider     text        NOT NULL CHECK (provider IN ('paypal','braintree','stripe','adyen')),
  sdk_version  text        NOT NULL,  -- 'jssdk-v5' | 'jssdk-v6' | 'web-sdk' | 'graphql' 等
  product_key  text        NOT NULL,  -- 叶子产品 slug，如 'spb-ecm' | 'acdc'
  display_name text        NOT NULL,
  description  text        NOT NULL DEFAULT '',
  enabled      boolean     NOT NULL DEFAULT true,
  sort_order   integer     NOT NULL DEFAULT 0,
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now(),
  UNIQUE(provider, sdk_version, product_key)
);

-- 自动更新 updated_at
CREATE TRIGGER demohub_products_updated_at
  BEFORE UPDATE ON demohub.products
  FOR EACH ROW EXECUTE FUNCTION public.update_updated_at();

-- RLS
ALTER TABLE demohub.products ENABLE ROW LEVEL SECURITY;
CREATE POLICY "public_read" ON demohub.products FOR SELECT USING (true);
-- 写入由 admin-console 通过 service_role key 操作，无需额外 policy
```

**admin-console 可操作字段：** `display_name`、`description`、`enabled`、`sort_order`
**只读字段（与代码路由绑定）：** `provider`、`sdk_version`、`product_key`

## Schema: `admin`

> admin-console 写入；各电商站启动时读取自己的支付渠道配置。

### 表：`admin.stores`

```sql
CREATE SCHEMA IF NOT EXISTS admin;

CREATE TABLE admin.stores (
  id           uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  store_type   text        NOT NULL UNIQUE,  -- 'fashion' | 'electronics' | 'ai-subscription' ...
  display_name text        NOT NULL,
  description  text        NOT NULL DEFAULT '',
  enabled      boolean     NOT NULL DEFAULT true,
  created_at   timestamptz NOT NULL DEFAULT now()
);

ALTER TABLE admin.stores ENABLE ROW LEVEL SECURITY;
CREATE POLICY "public_read" ON admin.stores FOR SELECT USING (true);
```

### 表：`admin.payment_channels`

```sql
CREATE TABLE admin.payment_channels (
  id           uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  store_type   text        NOT NULL REFERENCES admin.stores(store_type),
  provider     text        NOT NULL CHECK (provider IN ('paypal','braintree','stripe','adyen')),
  product_key  text        NOT NULL,  -- 具体支付产品 slug
  enabled      boolean     NOT NULL DEFAULT true,
  sort_order   integer     NOT NULL DEFAULT 0,
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now(),
  UNIQUE(store_type, provider, product_key)
);

ALTER TABLE admin.payment_channels ENABLE ROW LEVEL SECURITY;
CREATE POLICY "public_read" ON admin.payment_channels FOR SELECT USING (true);
```

**用途：** 每个电商站启动时读取 `WHERE store_type = 'fashion'`，决定展示哪些支付渠道及排序。

## Schema: `fashion`

> store-fashion 专属。未来每个电商站建立相同结构的独立 schema。

### 表：`fashion.profiles`

```sql
CREATE SCHEMA IF NOT EXISTS fashion;

CREATE TABLE fashion.profiles (
  id           uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid        NOT NULL UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name text,
  phone        text,
  avatar_url   text,
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now()
  -- UNIQUE(user_id) 自动创建索引，无需额外 CREATE INDEX
);

ALTER TABLE fashion.profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE fashion.profiles FORCE ROW LEVEL SECURITY;  -- 防止表 owner 绕过 RLS

-- ⚠️ 最佳实践：auth.uid() 用 SELECT 包裹，避免对每行调用（大表快 100x+）
CREATE POLICY "own_profile_all" ON fashion.profiles
  FOR ALL USING ((select auth.uid()) = user_id);
```

### 表：`fashion.orders`

```sql
CREATE TABLE fashion.orders (
  id               uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          uuid        NOT NULL REFERENCES auth.users(id),
  status           text        NOT NULL DEFAULT 'pending'
                               CHECK (status IN ('pending','paid','failed','refunded')),
  amount           numeric(10,2) NOT NULL,
  currency         text        NOT NULL DEFAULT 'USD',
  payment_provider text,       -- 'paypal' | 'braintree' | 'stripe' | 'adyen'
  payment_method   text,       -- 'paypal-button' | 'acdc' | 'apple-pay' ...
  payment_order_id text,       -- 第三方支付订单 ID（PayPal order ID 等）
  metadata         jsonb       DEFAULT '{}',  -- 弹性字段，存额外信息
  created_at       timestamptz NOT NULL DEFAULT now(),
  updated_at       timestamptz NOT NULL DEFAULT now()
);

-- ⚠️ Postgres 不自动为 FK 列建索引，必须手动创建
CREATE INDEX fashion_orders_user_id_idx ON fashion.orders(user_id);
CREATE INDEX fashion_orders_status_idx  ON fashion.orders(status);

ALTER TABLE fashion.orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE fashion.orders FORCE ROW LEVEL SECURITY;

-- ⚠️ 最佳实践：(select auth.uid()) 缓存为单次调用
CREATE POLICY "own_orders_all" ON fashion.orders
  FOR ALL USING ((select auth.uid()) = user_id);
```

## Auth 用户来源追踪

所有电商站共用 `auth.users`，通过 `user_metadata` 标记注册来源：

```js
// store-fashion 注册
await supabase.auth.signUp({
  email, password,
  options: { data: { store: 'fashion' } }
})
```

读取来源：
```js
const { data: { user } } = await supabase.auth.getUser()
const store = user.user_metadata.store  // 'fashion'
```

## 共享工具函数（`public` schema）

```sql
-- 所有 schema 的 updated_at trigger 共用此函数
CREATE OR REPLACE FUNCTION public.update_updated_at()
RETURNS TRIGGER AS $$
BEGIN NEW.updated_at = now(); RETURN NEW; END;
$$ LANGUAGE plpgsql;
```

## 扩展规则（新增电商站时）

新增 `store-electronics` 的步骤：

1. `CREATE SCHEMA electronics;`
2. 建立 `electronics.profiles`、`electronics.orders`（结构同 fashion）
3. 在 `admin.stores` 插入 `{ store_type: 'electronics', ... }`
4. 在 Supabase Dashboard 开启 `electronics` schema
5. 创建对应的 React app 和 Express 后端

## 命名规范总结

| 层级 | 规范 | 示例 |
|------|------|------|
| Schema | 小写，对应 app 简称 | `demohub`, `admin`, `fashion` |
| 表名 | 复数名词，snake_case | `products`, `orders`, `payment_channels` |
| 主键 | `id uuid DEFAULT gen_random_uuid()` | 统一 |
| 时间戳 | `created_at`, `updated_at` 均为 timestamptz | 统一 |
| 外键 | `{entity}_id` | `user_id`, `order_id` |
| 枚举值 | CHECK constraint 约束合法值 | `CHECK (status IN (...))` |

## Postgres 最佳实践（来自 supabase-postgres-best-practices）

### RLS 性能规范（CRITICAL）

所有 RLS policy 中的 `auth.uid()` 必须用 `SELECT` 包裹，避免对每行调用：

```sql
-- ❌ 错误：每行调用一次 auth.uid()（大表严重影响性能）
USING (auth.uid() = user_id)

-- ✅ 正确：调用一次并缓存（大表快 100x+）
USING ((select auth.uid()) = user_id)
```

### 外键索引规范（HIGH）

Postgres **不会**自动为外键列创建索引，必须手动添加：

```sql
-- 每张有 FK 的表都必须建索引
CREATE INDEX {table}_{fk_column}_idx ON {schema}.{table}({fk_column});
```

### FORCE ROW LEVEL SECURITY（CRITICAL）

用户数据表必须同时执行两条语句，防止表 owner 绕过 RLS：

```sql
ALTER TABLE {table} ENABLE ROW LEVEL SECURITY;
ALTER TABLE {table} FORCE ROW LEVEL SECURITY;  -- 缺这行，owner 可绕过
```

### 主键策略说明（HIGH）

本项目使用 `uuid DEFAULT gen_random_uuid()`（UUIDv4），原因：
- `auth.users.id` 是 UUID，外键兼容性要求保持 UUID 类型
- 权衡：可接受轻微 index fragmentation（数据量小的 demo/电商工具场景）
- 如未来数据量大，可迁移到 UUIDv7（需开启 `pg_uuidv7` 扩展）

### 连接池规范（CRITICAL）

- Supabase 内置 PgBouncer 连接池，使用**连接池 URL**（而非 direct URL）
- 应用代码中：使用 `SUPABASE_URL`（自动走连接池）
- 后台任务/migrations：用 direct URL（支持 DDL 语句）
- Supabase JS SDK 默认走连接池，无需额外配置
